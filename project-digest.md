## 1. Project Overview

**AlgDevs-AI** is a free-tier-first, full-stack autonomous AI agent platform. It is intended to evolve toward a Manus-style system: users chat with AI, create multi-step plans, spawn specialized agents, run code in a sandbox, and track artifacts in workspaces.

### Architecture

| Layer | Package | Technology |
|-------|---------|------------|
| Monorepo | root | pnpm workspaces, TypeScript 5.9 |
| Database | `@workspace/db` | PostgreSQL + Drizzle ORM + drizzle-zod |
| API Spec | `@workspace/api-spec` | OpenAPI 3.1 + Orval |
| Validation | `@workspace/api-zod` | Zod (generated from OpenAPI) |
| API Client | `@workspace/api-client-react` | React Query + custom fetcher |
| Backend | `@workspace/api-server` | Node.js 24 + Express 5 + esbuild |
| Frontend | `@workspace/algdevs-ai` | React 19 + Vite + Tailwind 4 + shadcn/ui |
| Sandbox | external | E2B REST API |
| AI | external | Groq (primary), Gemini, OpenRouter (fallback) |

### Deployment

- **Railway**: `railway.json` uses the Dockerfile and health-checks `/api/healthz`.
- **Render**: `render.yaml` uses the free plan with a hard-coded Neon Postgres URL.
- **Docker**: Multi-stage `Dockerfile` builds the monorepo and serves the API server from `dist/index.mjs` on port 8080.

## 10. Notable Patterns, Gaps, and Observations

- **OpenAPI-driven workflow**: The API is defined in `lib/api-spec/openapi.yaml`, then Orval generates both the Zod validators and the React Query client. This keeps the backend, frontend, and validation in sync.
- **Custom fetcher**: The frontend client supports base URLs and bearer tokens for React Native / Expo usage, even though the web app relies on cookies/same-origin.
- **Security awareness**: `pnpm-workspace.yaml` sets `minimumReleaseAge: 1440` (1 day) to mitigate supply-chain attacks; `apiKey` is stripped from provider responses.
- **Task queue**: The backend has a real PostgreSQL `SKIP LOCKED` task queue, but `executeTaskLogic` is only a stub. A real implementation would dispatch to the `supervisor`/`planner`/`coder` agents, call the AI layer, and write artifacts.
- **Plans**: Plans have steps, but plan execution (`POST /plans/:id/execute`) only flips status to `executing`; it does not actually run the steps. The step update endpoint updates `completedSteps` when a step is marked `completed`.
- **Sandbox**: Uses the E2B REST API directly (not the installed `@e2b/code-interpreter` SDK). Creates a sandbox, runs a command, deletes it in a `finally`.
- **Frontend is read-only for most entities**: It lists and creates tasks, plans, agents, etc., but does not yet implement editing, plan step management, or workspace editing.
- **Provider model**: Only Groq is fully configured via env var; Gemini and OpenRouter require user-provided API keys through settings. Fallback chain is hard-coded in `ai.ts`.
- **Database migrations**: No Drizzle migrations folder exists; `drizzle-kit push` is used (production cautious).
- **Hard-coded credentials**: `render.yaml` contains a real `DATABASE_URL` string pointing to a Neon Postgres instance.
- **PWA**: Complete manifest, service worker, icons, and mobile meta tags are present.
